# OpenRing — Doorbell Firmware & Streaming

This document is the design spec for the **device-side** software (the bit
that runs on the doorbell, not on the host computer). It is intentionally
opinionated — the v0.1 path is one specific choice; alternatives are
captured but not built.

## Goals

1. The doorbell is a **thin appliance.** All inference, all storage, all
   business logic lives on the host. The doorbell only:
   - serves a live H.264 RTSP stream
   - reports button presses
   - reports a periodic heartbeat so the host can show "doorbell offline"
   - (v0.3+) bridges audio in both directions
2. The doorbell is **replaceable** — swap the SD card, run one script,
   you're back online. No state lives on the device that matters.
3. Boot-to-streaming under 60 seconds on a cold power-up.
4. Power source is whatever's behind your existing chime button (16-24 V
   AC) or a 5 V USB supply, your call. No batteries — the power draw of
   real-time H.264 + Wi-Fi makes battery operation a dead end on this
   class of hardware.

## Reference hardware (v0.1)

| Part | Role | ~USD |
|---|---|---|
| Raspberry Pi Zero 2 W | SoC + Wi-Fi | $15 |
| Pi Camera Module 3 (wide, no IR filter) | imaging | $35 |
| 32 GB SanDisk Industrial microSD | OS + MediaMTX cache | $12 |
| 12 mm momentary panel-mount button | the actual doorbell button | $5 |
| 16-24 V AC → 5 V/3 A buck converter (e.g. Mean Well IRM-15-5) | power off existing chime transformer | $12 |
| 3D-printed weatherproof shell | enclosure | bring your own filament |
| **Total** | | **~$80** |

For audio (deferred to v0.3): a USB-C-to-3.5 mm DAC + a small electret
mic and 1 W speaker, ~$20 more.

The **Pi Zero 2 W** is the sweet spot: ARMv8 quad core, hardware H.264
encode in `libcamera-vid`, full Linux means we can drop in MediaMTX
without writing a streaming server ourselves. The next tier down
(ESP32-S3 + OV2640) saves $10 but cannot do H.264, audio is painful, and
you end up writing a lot of bespoke firmware to replace what apt-get
gives you for free. Not worth it for v0.1.

The next tier up (Pi 5 + Hailo accelerator at the door, doing on-device
inference) doubles the cost and breaks the "all the smarts on the host"
architectural premise. Reconsider after v1.0 if there's demand for an
"isolated" mode where the doorbell still notifies via a local push
service when the host is down — but that's a different product.

## Streaming stack

| Layer | Choice | Why |
|---|---|---|
| Camera capture | `libcamera-vid` → H.264 NALU pipe | hardware encoder on the Pi, no userspace re-encode |
| Streaming server | [MediaMTX](https://github.com/bluenviron/mediamtx) | pure Go, pulls from a pipe, serves RTSP/RTMP/HLS, ~12 MB binary, zero config beyond a YAML file |
| Wire format | RTSP/H.264 over the LAN | same protocol the host's `detector` already speaks (the host stack is forked from a project that consumes UniFi RTSP) |
| Live view in browser | MJPEG snapshot poll on port 8081 (v0.1), upgrade to WebRTC via MediaMTX's WHEP endpoint (v0.4+) | MJPEG is dirt simple and bandwidth is fine on a LAN; WebRTC is the upgrade path when we want sub-second latency for two-way audio |

The detector on the host opens
`rtsp://openring-doorbell.local:8554/door` exactly the same way ScarGuard
opens a UniFi camera URL. **Zero code changes** in the host detector —
that's the point of choosing RTSP over a bespoke protocol.

`config/mediamtx.yml` (committed to the repo, copied to the Pi during
provisioning) looks roughly like:

```yaml
paths:
  door:
    runOnInit: |
      libcamera-vid -t 0 --inline -o - --width 1280 --height 720 \
        --framerate 15 --codec h264 --profile high --intra 30 \
        | ffmpeg -f h264 -i - -c copy -f rtsp rtsp://localhost:$RTSP_PORT/door
    runOnInitRestart: yes
authMethods: [basic]
authInternalUsers:
  - user: openring
    pass: ${RTSP_PASSWORD}
    permissions:
      - action: publish
        path: door
      - action: read
        path: door
```

Auth uses HTTP Basic with a per-deployment password (generated by
`pi-setup.sh`, surfaced once, stored in `/etc/openring/secrets.env`
mode 600). The host's detector reads the same password from a Docker
secret. **Both sides must verify TLS** (or stay on the LAN) — we
explicitly don't ship a "convenience mode" that disables this.

## Doorbell button

A single GPIO input. v0.1 does the simplest thing that works:

```
GPIO 17 (with internal pull-up) ─── button ─── GND
```

A small Python service (`button.py`) uses `gpiozero`:

```python
from gpiozero import Button
button = Button(17, pull_up=True, bounce_time=0.05)
button.when_pressed = on_press
```

`on_press` does **one** HTTP `POST` to the host:

```
POST https://openring-host.local/api/doorbell/press
Authorization: Bearer <pi-token>
Content-Type: application/json

{"timestamp": "2026-05-09T18:42:11Z", "device_id": "front-door"}
```

The Bearer token is provisioned during `pi-setup.sh` and stored under
`/etc/openring/secrets.env`. The host validates it like any other API
token and re-publishes the event onto its internal Redis bus
(`openring:doorbell`) with an HMAC signature, where the notifier and web
SSE feed pick it up.

We deliberately do **not** put Redis on the Pi or have the Pi publish
directly to the host's Redis. Reasons:

1. The Pi only ever has *one* Redis client of its own to maintain — none.
   Every dependency we ship to the device is one we have to keep
   patched.
2. The host is the security boundary. The Pi authenticates against a
   single HTTPS endpoint; the host decides what gets onto the bus.
3. Network failure modes are simpler to reason about: a button press is
   a single retried POST with idempotency key (`device_id + timestamp`),
   not a queued pub/sub message.

The button POST is **independent of motion detection.** A press always
fires a notification, even if no person is detected on camera (someone
small, an obscured lens, a delivery driver who pressed and ducked). This
mirrors how a real doorbell works — the button is the source of truth
for "someone deliberately wanted your attention."

## Heartbeat

Same Python service does a lightweight POST every 60 s:

```
POST https://openring-host.local/api/doorbell/heartbeat
{
  "device_id": "front-door",
  "uptime_seconds": 184201,
  "rtsp_clients": 1,
  "cpu_temp_c": 52.4,
  "wifi_rssi_dbm": -67,
  "version": "0.1.0"
}
```

If the host hasn't seen one in `90 s` it surfaces "Doorbell offline" on
the dashboard and (configurably) fires a notification. This re-uses the
same camera-health-monitoring pattern from ScarGuard
(`services/web/src/.../camera_health.py` in the upstream).

## Audio (v0.3+, deferred)

Just enough detail to confirm the architecture leaves room for it.

| Direction | Pi side | Host side |
|---|---|---|
| Mic → web | `arecord -f S16_LE -r 16000 -c 1` → opus encode → WebSocket frames to host | host relays frames to the browser via the live-view WebSocket; browser plays via `MediaSource` |
| Web → speaker | browser captures mic → opus over WebSocket → host → WebSocket to Pi → `aplay` | push-to-talk button gates capture |

The Pi runs a small `audio-relay` Python service that owns a single
WebSocket to the host. We will not run a full SIP/WebRTC stack on the
Pi Zero 2 W — the CPU budget is tight and we don't need the
full-duplex echo cancellation that consumer doorbells offer. Push-to-
talk half-duplex is fine for v0.3.

WebRTC for live view (using MediaMTX's WHEP endpoint) is the natural
v0.4 upgrade — it gives us the bidirectional path "for free" using the
same protocol stack the browser already has.

## Provisioning (`pi-setup.sh`)

The on-Pi installer is a single bash script
(`services/doorbell-firmware/pi-setup.sh`) you run after flashing
Pi OS Lite 64-bit. It does, in order:

1. **Sanity checks.** Confirms it's running on a Raspberry Pi (warns
   otherwise) and resolves a supported architecture
   (`aarch64` → `arm64`, `armv7l` → `armv7`, `x86_64` → `amd64` for
   dev). Refuses to run as non-root unless `--dry-run` is passed.
2. **`apt-get install`** of `python3`, `python3-gpiozero`,
   `python3-requests`, `libcamera-apps`, `ffmpeg`, `iw`, `curl`, `jq`,
   `ca-certificates`, `gettext-base`. All idempotent.
3. **`openring` user + directories.** Service user with `gpio` and
   `video` groups; `/opt/openring`, `/var/lib/openring`, `/etc/openring`.
4. **Pinned MediaMTX install.** Downloads
   `mediamtx_v<version>_linux_<arch>.tar.gz` from the GitHub release,
   verifies SHA256 against `services/doorbell-firmware/mediamtx.sha256`
   in this repo (which carries the upstream-published checksums for
   the pinned version), installs the binary to `/usr/local/bin`.
   Skips the download if the right version is already present.
   `--skip-hash` overrides verification (development only).
5. **Pairing.** Auto-generates a 32-char RTSP password (overridable
   via `--rtsp-password`) and POSTs `/api/doorbell/register` to the
   host. The endpoint is bearer-less but gated on a 5-minute pairing
   window the operator opens from the web UI; if the window is closed
   the script exits with a clear error pointing back at the UI step.
   The response carries a fresh device token that's immediately
   written to disk and never logged.
6. **`/etc/openring/secrets.env`** (root:openring 0640) carrying
   `HOST_BASE_URL`, `DEVICE_ID`, `DEVICE_TOKEN`, `RTSP_PASSWORD`,
   `VERSION`. Read by every device-side service via
   `common.load_settings()`.
7. **`/etc/openring/mediamtx.yml`** rendered from the in-repo
   template via `envsubst`, with the RTSP password substituted in.
8. **systemd units installed** — `openring-mediamtx.service`,
   `openring-button.service`, `openring-heartbeat.service` —
   `daemon-reload`ed and `enable --now`ed. The script reports each
   unit's `is-active` state on completion.
9. **Output the camera YAML snippet** the operator pastes into
   `openring.yml` on the host so the detector can pull this Pi's
   stream:
   ```yaml
   - name: front-door
     rtsp_url: "rtsp://openring:<password>@<pi-hostname>:8554/door"
     enabled: true
     resolution: 720
     notification_rules:
       - class_name: doorbell_press
         channels: [phone-ntfy]
       - class_name: person
         channels: [phone-ntfy]
   ```

The whole thing takes under two minutes on a stock Pi. The only secret
that ever leaves the device is the one-time pairing handshake — after
that, every call is a Bearer-authenticated POST. Re-running the script
during a fresh pairing window rotates the device token (host-side this
is an `INSERT-or-UPDATE` on `device_tokens`); the device's identity
in the host's registry is preserved.

`--dry-run` prints every step without touching the system or making
the network call — useful for validating arguments before committing.

CI runs the script with `bash -n`, `--help`, and `--dry-run` plus
shellcheck on every PR.

## Failure modes we explicitly handle

| Failure | Behavior |
|---|---|
| Host unreachable | Button service queues up to 16 press events on disk, replays when the host comes back, drops oldest beyond that |
| RTSP stream dies | systemd `Restart=always` plus MediaMTX's own `runOnInitRestart: yes`. Detector on the host auto-reconnects with backoff (already supported in the inherited stack). |
| Wi-Fi flap | systemd-networkd handles the reconnect; button service's POST retries with exponential backoff, capped at 60 s |
| SD card read-only after a power loss | systemd `ConditionPathIsReadWrite=/var/lib/openring`; if not, the service refuses to start and writes to the journal — the host detects via missing heartbeat and notifies |
| Camera unplugged / dead | `libcamera-vid` exits, MediaMTX `runOnInit` restart loop hits a backoff, host detector logs RTSP failure, host fires "camera offline" notification |
| Clock skew on the Pi | We send timestamps in UTC ISO 8601 from the Pi but the host **always** stamps event arrival with its own clock. The Pi's clock is for telemetry only. |

## Failure modes we deliberately don't handle in v0.1

- The host being completely off. The doorbell is dumb without a brain.
  This is a documented limitation; "isolated mode" with a local chime
  buzzer is on the v1.x list.
- Battery operation. Not pursued.
- Multiple cameras on one Pi. The MediaMTX config supports it, but we
  don't surface it in `pi-setup.sh`. Power users can edit the YAML.

## Deliberate non-features

- **No mDNS-based auto-discovery from the host to a freshly-imaged Pi.**
  Discovery is a foot-gun on multi-tenant networks (apartment Wi-Fi,
  shared offices). Pairing is an explicit human action.
- **No cloud relay for NAT traversal.** If you want remote access, run
  Tailscale or your own VPN. We won't ship a UDP hole-punching service.
- **No firmware OTA from us.** The Pi pulls updates via `apt-get` and
  the version-pinned MediaMTX hash in this repo. There is no signed
  binary blob we push to your device. (`pi-setup.sh --update` is the
  manual upgrade path.)

## What lives in the repo

```
services/doorbell-firmware/
├── pi-setup.sh                  # the installer (idempotent)
├── mediamtx.sha256              # SHA256 hashes for the pinned MediaMTX release
├── requirements.txt             # Python deps (gpiozero, requests)
├── src/
│   ├── button.py                # GPIO button → host POST, on-disk retry queue
│   ├── heartbeat.py             # periodic telemetry POST
│   └── common.py                # shared HTTP client w/ retry + auth
├── systemd/
│   ├── openring-mediamtx.service
│   ├── openring-button.service
│   └── openring-heartbeat.service
├── config/
│   └── mediamtx.yml.template    # rendered by pi-setup.sh via envsubst
└── tests/                       # pytest, GPIO mocked via gpiozero's MockFactory
    ├── test_button.py
    └── test_common.py
```

All Python on the device targets Python 3.11 (Pi OS Bookworm default).
Tests run on the host's CI using `gpiozero`'s mock-pin backend, no real
hardware needed.
